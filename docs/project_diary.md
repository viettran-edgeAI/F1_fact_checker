# Project Diary

- 2026-06-13: Web evidence needed a second pass beyond Brave snippets. Fetching readable article text for each normalized top candidate and preserving body text through compaction was necessary to keep Gemma from seeing title-only evidence. The larger `LLM_CTX_SIZE` in Docker Compose gives that richer packet enough room.
- 2026-06-13: A structured-only rewrite stage was too narrow for the real pipeline. Generalizing it into claim context completion before both retrieval routes keeps abrupt web claims anchored in their source text while still letting structured claims refresh local query text when the completion changes.
- 2026-06-14: Gemma 4 MTP is a runtime acceleration feature, not a prompt or verdict-behavior feature. Keep it behind explicit `LLM_MTP_ENABLED` and draft-model env vars. The target model is `gemma-4-E2B-it-qat-UD-Q4_K_XL.gguf` with the dedicated MTP drafter. The rebuilt llama.cpp `9625` runtime loads the drafter; on the Jetson full-stack profile, MTP needs `LLM_CTX_SIZE=12288`, `LLM_BATCH_SIZE=512`, `LLM_UBATCH_SIZE=128`, `LLM_SPEC_DRAFT_N_MAX=2`, and `GGML_CUDA_DISABLE_GRAPHS=1` to avoid CUDA allocation failures while OCR is resident.
- 2026-06-14: MTP tuning is workload-specific. A long free-form text prompt favored `LLM_SPEC_DRAFT_N_MAX=1`, but the actual final verdict JSON prompt favored `2` and matched the observed 45-54 tok/s site range. Tune against the displayed final-step prompt, not arbitrary generation prompts.
- 2026-06-14: Compose may start GPU services concurrently even when health dependencies exist for downstream services. For the MTP profile, startup ordering is part of the memory budget: load `llm-service` first, then start OCR after LLM is healthy.
