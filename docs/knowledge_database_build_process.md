# F1 Knowledge Database Build Process

## 1. Goal

The local knowledge database is built to provide reliable evidence for the F1 fact-checking system.

Its role is to support claims such as:

* who won a race
* driver and constructor standings
* race results
* championship winners
* qualifying results
* season-level statistics
* historical Formula 1 facts

The database is mainly used as the local evidence source before Gemma generates the final verdict.

---

## 2. Main Data Sources

The knowledge database is built from two sources:

### Formula 1 World Championship Dataset

This dataset provides historical F1 data from 1950 to 2024.

It contains structured tables such as:

* races
* drivers
* constructors
* results
* qualifying
* driver standings
* constructor standings
* circuits
* seasons

This source is suitable for stable historical facts.

### Jolpica F1 API

Jolpica F1 API is used to update or supplement the local dataset.

It is useful for:

* fetching newer season data
* checking updated results
* extending the local database beyond the static dataset
* keeping the knowledge base maintainable over time

---

## 3. Build Pipeline Overview

The dataset build process should follow this flow:

```text
Raw F1 data
↓
Normalize data
↓
Convert structured rows into natural-language facts
↓
Store facts and metadata
↓
Generate embeddings
↓
Build FAISS index
↓
Use retrieved evidence for Gemma verdict generation
```

The key design decision is:

```text
Do not embed raw table rows directly.
Convert each structured row into short natural-language facts first.
```

---

## 4. Data Normalization

Before generating facts, raw data should be cleaned and normalized.

Normalization includes:

* mapping IDs to readable names
* standardizing driver names
* standardizing constructor names
* standardizing race names
* standardizing circuit names
* normalizing season/year fields
* handling missing values
* removing duplicate records
* merging data from the static dataset and Jolpica API

Example:

```text
driver_id: max_verstappen
→ Max Verstappen
```

```text
constructor_id: red_bull
→ Red Bull Racing
```

```text
race name: Abu Dhabi GP
→ Abu Dhabi Grand Prix
```

The goal is to make every fact consistent before it is embedded.

---

## 5. Natural-Language Fact Generation

Most F1 data is stored as structured rows, but the retrieval model works better with natural-language text.

Instead of embedding a row like this:

```text
2023 | Abu Dhabi Grand Prix | Max Verstappen | Red Bull | 1 | 25
```

the system should generate facts like:

```text
Max Verstappen won the 2023 Abu Dhabi Grand Prix.
Max Verstappen finished 1st in the 2023 Abu Dhabi Grand Prix.
Max Verstappen scored 25 points for Red Bull Racing in the 2023 Abu Dhabi Grand Prix.
```

This makes FAISS retrieval more useful for natural claims such as:

```text
Verstappen won Abu Dhabi in 2023.
The 2023 Abu Dhabi GP was won by Max Verstappen.
Max won the final race of the 2023 season.
```

The fact generation should be deterministic and template-based, not generated by Gemma.

Gemma should not be used to rewrite every row because the source data is already structured. Template-based generation is faster, more stable, and easier to verify.

---

## 6. Suggested Fact Types

The first version should focus on common, high-value fact types:

```text
race_winner
race_result
race_points
driver_champion
constructor_champion
driver_standing
constructor_standing
qualifying_result
pole_position
driver_constructor_relationship
```

Each fact should be short and focused on one meaning.

For example, from one race result row, the system may generate several facts:

```text
Max Verstappen finished 1st in the 2023 Abu Dhabi Grand Prix.
Max Verstappen won the 2023 Abu Dhabi Grand Prix.
Max Verstappen scored 25 points in the 2023 Abu Dhabi Grand Prix.
Max Verstappen drove for Red Bull Racing in the 2023 Abu Dhabi Grand Prix.
```

This improves recall because different user claims may focus on different aspects of the same row.

---

## 7. Metadata Storage

Each generated fact should keep metadata from the original source.

Example:

```json
{
  "fact_type": "race_winner",
  "fact_text": "Max Verstappen won the 2023 Abu Dhabi Grand Prix.",
  "source": "Jolpica F1 API",
  "season": 2023,
  "race_name": "Abu Dhabi Grand Prix",
  "driver_name": "Max Verstappen",
  "constructor_name": "Red Bull Racing"
}
```

Only `fact_text` is embedded.

The metadata is used later to:

* display evidence clearly
* trace the source of the fact
* filter facts if needed
* help Gemma explain the verdict

---

## 8. Embedding and FAISS Indexing

After fact generation, each `fact_text` is converted into an embedding using:

```text
~models/embedding/all-MiniLM-L6-v2
```

The embeddings are stored in FAISS for semantic retrieval.

The index build process should use small batches to avoid RAM spikes, especially on Jetson Orin Nano.

Recommended approach:

```text
facts
↓
embed in small batches
↓
write vectors incrementally
↓
build FAISS index
↓
store mapping between FAISS vector IDs and fact IDs
```

This avoids loading the entire corpus into memory at once.

---

## 9. Runtime Retrieval Flow

At runtime, the fact-checking pipeline works like this:

```text
User input
↓
Gemma extracts claims
↓
Each claim is embedded
↓
FAISS retrieves the most relevant facts
↓
The system loads fact metadata
↓
Gemma receives claim + retrieved evidence
↓
Gemma generates verdict and explanation
```

Example:

```text
Claim:
Lewis Hamilton won the 2021 Abu Dhabi Grand Prix.

Retrieved evidence:
- Max Verstappen won the 2021 Abu Dhabi Grand Prix.
- Lewis Hamilton finished 2nd in the 2021 Abu Dhabi Grand Prix.

Expected verdict:
False.
```

---

## 10. Role of Gemma

Gemma should be used for:

* extracting claims from user input
* interpreting retrieved evidence
* generating verdicts
* writing explanations

Gemma should not be used for:

* generating the local fact corpus from structured rows
* rewriting every dataset row
* acting as the primary database layer

The knowledge database should remain deterministic and reproducible.

---

## 11. Design Summary

The knowledge database should be built as follows:

```text
F1 World Championship Dataset + Jolpica F1 API
↓
Normalize structured data
↓
Generate short natural-language facts using templates
↓
Store facts with metadata
↓
Embed fact_text with all-MiniLM-L6-v2
↓
Build FAISS vector index
↓
Use retrieved facts as evidence for Gemma
```

The most important point is:

```text
The dataset remains structured internally,
but the retrieval layer uses natural-language facts.
```

This keeps the system simple, consistent, and suitable for a RAG-based F1 fact-checking pipeline.
